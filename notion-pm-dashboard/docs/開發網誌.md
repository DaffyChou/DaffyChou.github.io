# 用 Claude × Cowork × Notion MCP 打造專案管理整合儀表板

> 一個 PM 從 58 個 Notion 專案、2,200 多筆任務中,如何用 LLM agent 在一天內生出可即時刷新的視覺化儀表板。

**作者**:Daffy Chou × Claude (Cowork mode)
**日期**:2026-05-08
**最終交付**:單檔 HTML Live Artifact(539KB,內含 11 專案 / 306 任務 / 25 修正項目)

---

## 1. 為什麼要做這個

公司用 Notion 管理專案,但 Notion 天生不是 BI 工具。一個 PM 同時盯 58 個專案,沒辦法做到下面這些事:

- 一眼看出哪些專案逾期、誰超載、哪些有未結 bug
- 跨專案彙總任務狀態與工時
- 自動辨識「該介入」的專案(風險評分)
- 給高層一份不需要點開 50 個頁面的視覺報告

**核心痛點**:Notion 強在儲存與協作,但缺彙總視角。我需要把資料抽出來、計算指標、視覺呈現,而且要能即時刷新。

## 2. 架構決策

```
Notion (3 個資料庫)
  ├─ 專案 (58 筆)
  ├─ 任務 (~2,200 筆)
  └─ 修正清單 (Bug)
        ↓ Notion MCP (notion-query / notion-fetch)
Python pipeline (本機,一次性)
  ├─ 清理 JSON 編碼字串
  ├─ 解析 user://、relation://、formulaResult://
  ├─ 計算 KPI / 工作量 / 風險分數 / 後續建議
  └─ 嵌入 HTML(JSON.parse 反解轉義)
        ↓
單一 HTML Live Artifact (539KB)
  ├─ Chart.js (CDN, doughnut/bar)
  ├─ Grid.js (CDN, 可搜尋表)
  ├─ 內嵌 dashboard_data (JSON)
  └─ window.cowork.callMcpTool 即時刷新
        ↓
24h localStorage 快取
```

**為什麼是 Live Artifact 不是純靜態 HTML?**
- 客戶端透過 `window.cowork.callMcpTool` 可以重新拉 Notion 最新資料
- 不需要重跑 Python pipeline,使用者按一個按鈕就能看到最新狀態
- 過 24h 快取自動失效,避免看到過期資訊

## 3. 七大功能區塊

| # | 區塊 | 內容 |
|---|---|---|
| 1 | **KPI 卡片** | 專案總數、完成率、高/中風險、進行中/尚未開始/逾期任務 |
| 2 | **任務分組樹 + Grid.js 搜尋表** | 依專案分組(只顯示活躍任務,隱藏暫停與完成);全任務可搜尋表 |
| 3 | **人員工作量** | 堆疊長條圖 + 表格(顯示主要專案、佔比、逾期數) |

| 5 | **待追蹤任務 Top 30** | 逾期/修正中/測試卡關/高優未完成 - 按逾期天數排序 |
| 6 | **風險分析** | 8 條規則加總評分:高(≥5) / 中(≥3) / 低(≥1) / 安全 |
| 7 | **後續建議** | 依風險自動產出 5-7 條具體可執行建議 |

### 風險評分演算法
```
+3 結案需優化
+3 合約已逾期
+2 暫停 / 合約 30 天內到期 / 無負責人
+1 積壓有任務 / 未結 bug>5 / 任務>50
```

完全機械化,不依賴主觀判斷。

## 4. 互動操作(P1)

- **狀態 chip 篩選**:積壓 / 進行中 / 暫停 / 結案需優化…可單獨切換,選擇存 localStorage
- **🔄 重新拉取**:MERGE 模式,保留 bulk 任務並用最新版覆蓋
- **⚡ 全量載入**:批次 fetch 所有任務(10 並發),完成後 24h 快取

- **🗑️ 清除快取**:清空 localStorage 並重整
- **🔬 診斷模式**:測試 callMcpTool 回應,alert 顯示原始格式與解析結果

## 5. 技術亮點 — 五個踩坑與解法

### 踩坑 1:Notion API 單次 100 筆上限
`notion-query-database-view` 一次最多 100 筆,且 `has_more: true` 時無法分頁(MCP wrapper 沒實作 cursor)。

**解法**:每個 Notion 專案頁面有自己的 inline tasks database,內建「where 專案 = 此專案」的 view 過濾器。直接 query 那個 view 等於拿到單一專案的全部任務。比逐筆 `notion-fetch` 任務 URL 快 50 倍。

```javascript
// 中信構型文件管理平台的專屬任務 view
'https://www.notion.so/1c31e8e6b4828056bf34d7d8d46c1e69?v=1c31e8e6-b482-80ef-99c1-000c3797a65f'
```

### 踩坑 2:Write 工具有 ~44KB 寫入上限
寫入超過 44KB 的 JS 檔會被截斷在中間,造成語法錯誤。

**解法**:改用 `mcp__workspace__bash` 內的 Python 直接寫入,不經過 Write tool。同時:檔案被截斷後常遺留 NULL bytes,需用 Python 掃描清理:
```python
data = open('script.js','rb').read()
end = len(data)
while end > 0 and data[end-1] == 0:
    end -= 1
clean = data[:end].rstrip() + b'\n'
```

### 踩坑 3:callMcpTool 回應 4 種形狀
不同情境下 MCP 回傳格式不一致:
- 直接物件 `{text: "..."}`
- 包在 `{content: [{type, text}]}`
- 字串
- 陣列 `[{type, text}]`

**解法**:寫 `extractFetchText(r)` 統一處理 4 種形狀,前端做 schema agnostic。

### 踩坑 4:Cache 污染
之前 session 的 localStorage 快取了非白名單專案的任務,重新打開時這些任務會「復活」並出現在儀表板。

**解法**:三重防呆白名單過濾:
1. 初始載入 `applyWhitelistFilter()`
2. 快取還原 `tryRestoreCache()` 同步過濾
3. 每次 `recomputeAggregates()` 重新檢查

```javascript
DATA.tasks = DATA.tasks.filter(t =>
  t.project_ids.some(pid => WHITELIST_PIDS.has(pid))
);
```

### 踩坑 5:Chart.js CDN 載入時序
`init()` 在 DOMContentLoaded 觸發,但 Chart.js CDN 此時可能還沒下載完。

**解法**:三段式 bootstrap:
- DOMContentLoaded 嘗試一次
- window.load 後備觸發
- tryRender 輪詢 30 秒(等待 Chart.js 出現)
- 即使逾時也渲染表格資料,只是圖表 canvas 為空

## 7. 開發過程觀察

### LLM agent 寫前端的可行性
從 0 開始建構一個 539KB 的 HTML/JS 應用、12 個迭代調整、5 次重大架構修改 — 全部由 Claude (Cowork mode) 在一個 session 內完成,我只負責提需求與驗收。可行性已經超過想像,**但需要 10+ 輪迭代是常態**。

### 「probe before build」是黃金法則
MCP wrapper 與底層 API 的 schema 經常不同。在嵌入解析邏輯之前,**先實際呼叫一次工具看真實回應**,比讀文件可靠 10 倍。我為此寫了「🔬 診斷模式」按鈕,專門幫使用者回報 callMcpTool 真實格式。

### 工具有限制,但繞得過去
- Write 上限 → 改 bash python
- Edit 會留 null bytes → 寫腳本清理
- API 100 筆上限 → 換成 inline view
- 4 種回應格式 → 用 adapter pattern 統一

每個工具都有它的痛,但都不是 blocker,只是會讓開發路徑變長。

### 把 LLM agent 當「資深前端 + 資料工程師」用
給它需求、它給你解法。但別期望它一次到位 — 你要扮演 PM 角色,告訴它優先級、驗收標準,然後不斷小迭代。每輪不超過 3 個改動,讓它有空間針對性處理。
